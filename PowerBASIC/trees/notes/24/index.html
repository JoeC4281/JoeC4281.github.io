      There is a lot of stuff to consider in writing a compiler. The steps are broken down into things like the parser, linker, etc. I don't think you have to follow the so called "official" methods for compiler design, but you still have to tackle the basic tasks of parsing the code into something you can use for code generation, generate the code (either directly to machine language) or use a backend like C (convert from basic to C) or assembler.

      Many big name companies today are using LLVM, which is kind of a step up from assembler (a pseudo language), for the backend since it can compiler to different target platforms (ie. x86 or ARM). Personally I would use LLVM rather than assembler for a backend.

      https://llvm.org/

      Take a look at who uses LLVM:

      https://llvm.org/Users.html

      Such as Adobe, Apple, NVidia and others.

      Shop around for books about writing compilers. Sadly most such books are not well written for first timer compiler writers. I have a bunch but haven't dug into most of them yet.

      I also am interested in writing not only a compiler, but language parsers (ie. convert BASIC to other languages).

      It's been a long time since I tried my hand at writing a compiler. Back in the 1980's I wrote my own compiler for the Commodore 64. I used Abacus BASIC (a BASIC compiler) to write my own low level language which I would say was minimal BASIC, more like high level Assembler/Basic hybrid. You need a programming language which works well with strings. That is why I used Abacus Basic. In my case I generated directly to 6502 machine language. Fortunately 6502 machine language is very simple and there was an excellent book on it which helped me. I then used my compiler to write a simple family friendly video game:

      http://www.gamebase64.com/game.php?id=1465&d=45

      The game was published in the 1987 Compute Gazette magazine:

      http://gazettegalore.blogspot.com/20...ot-schnip.html

      The game was fun, but simple. The real experience was that I wrote my own compiler so I could write the game. The performance of the game was excellent since it was compile directly to machine code and was optimized for minimal overhead. The compiler too was very simple. For example you could only have 26 variables and the names were predefined with the letters A to Z (one character). The variables were predefined in a memory table.

      While I don't think I would write my own compiler if I were trying to support a full language like Powerbasic, I do think it may be useful to create a language which has some special function such as IOT (Internet of Things) or Windows Core or something like that. One would add features specific to the unique needs of the platform or device one is targeting.

          Gary,

          Don't waste your time on BASM.

          You would be better off playing with BCX.

          Some people disparage source-to-source translators as not being real compilers. Some people are stupid. And there are some people who have learned the value of peer reviewed, open,standards compliant libraries and source code. That's pretty important to me.

          The code that BCX emits can be compiled and linked (32/64) using a half dozen or more Windows compilers. Using two (or more) compilers on the same source code is a great way to catch problems during development. I personally compile using LccWin32, Pelles C, Mingw, MSVC++, and CLANG before I upload any new release of BCX. This year, I improved BCX so that the source code it emits compiles using the LLVM / Clang compiler which is renown for its superior code quality, compatibility, helpful compiler warnings and error messages. If you don't know about LLVM / Clang, this 2 year old article is just a teaser for you.

          https://arstechnica.com/gadgets/2018...ow-uses-clang/

          Google, Apple, Facebook, and lots of other companies have invested in LLVM. There must be a reason for that.

            I don't want to rain on the parade but approaching a 64 bit compiler stone cold with no experience is a task that is probably beyond you. The calling convention is different from Win32 and complies with the Microsoft ABI (Application Binary Interface). Then you have to construct all of the import tables in the 64 bit PE header then generate the binary code for the instruction set of the chosen language form and do this with correct assembler instructions.

            One alternative is to use a back end assembler, GCC uses the GNU assembler GAS, it can be done with a low level assembler like FASM or one of the recent Watcom derivatives like UASM or JWASM but you will need to learn a massive amount of 64 bit assembler to do this and few have the experience to do this.

                      Suppose I wanted to write a minimal 64 bit compiler, written in assembly language
                      First, learn assembly language.

                      FWIW, you can write a compiler in any language. The main output is the same thing: executable code. It makes no difference if the output is 16 bit or 32 bit or 64 bit or anything else. The source code parsing and translation to the desired 'opcodes' is the same.

                      That is, there is no reason you cannot write a compiler using 32-bit PowerBASIC with an output of 64-bit Intel-compatible executable code. And unless I am mistaken, PowerBASIC's first Win/32 executable output compilers were themselves 16-bit executable programs!

                                Hi Gary,

                                While you learn the ins and outs of making a compiler I would suggest that you do that with PB as you already have considerable experience with using it.

                                You will no doubt learn much assembler on the way.

                                Dont be put off by people saying it is a large task.
                                You have much of what you need already. You have made a calculator which needed to evaluate expressions both as a calculator and for plotting.
                                You need some form of parsing and you have handled that at least in part in your gbThreads and other applications.

                                You may wish to advance your parsing to the NFA DFA approach (automata coding) but using PARSE$ works well too.

                                There was already code on this forum for a "Recursive Descent Compiler" provided by Randall Hyde.
                                And also for creating a parser using BNF. (Backus Naur Form)

                                A Good beginners book with code in classic VB and VB NET would be "Build Your Own ,NET Language and Compiler" available in ebook form and the code is available from the apress website.
                                On checking my library (several thousand volumes) I have a dead tree copy of that book which I could post to you.

                                I also have a large array of ASM source code for maths and other data types not native to PB. I have Intel, Arm, PIC32 and other processor versions. I had offered to PB but there was no interest. If there is something you have difficulty with while creating your compiler, feel free to post a message and I will try to help.

                                As another approach, ROSLYN, the Visual Studio Compiler is open source now and supports the compiler as a service. mmm PB.NET anyone.
                                No No No... sorry .NET is all the slow bloated indirection that PB is not... I will go and flog myself now for even jokingly suggesting it.

                                While writing a direct compiler competitor in PB may be against the licensing, creating a DSL (Domain Specific Language) compiler should not be.
                                DSL Tools in PB would be good to have.

                                I have PB code for table scanning which I can post if you want.

                                The basics for starting are SCANNING, PARSING, and OBJECT Code optimisation, RUNTIME Code emmission.
                                Many Optimisations are only possible from carefully created Syntax Trees so you may want to think about the format for your syntax trees which is normally the output of your PARSER.

                                If you want to get started with a compiler but are concerned with the PE headers and linking of DLL's (you were going to do that right), then consider leaving it to an outside program.
                                With PB we are spoilt by the fact that it includes a linker where other compilers do not. MS LINK can be used and is available free with a community edititon install of VS2019.
                                Documentation for the MS Linker.EXE can be found here: https://docs.microsoft.com/en-us/cpp...g?view=vs-2019

                                For a different approach try a youtube search. Here is a link for creating a 64bit compiler from scratch(so it says). https://www.youtube.com/watch?v=N9B2KeAWXgE

                                The source code is here: https://gist.github.com/pervognsen/3...dbd35d88c4f323


                                Don't be put off. The pitfalls are all part of learning and form the FUN which we as developers enjoy.
                                Kind regards,
